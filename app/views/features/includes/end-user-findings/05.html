In our concept test, it was not initially clear to users what legacy data was, or what it would enable.

The description of legacy data presented a content design challenge. In our prototype we described legacy data as ‘past data’ however this was unclear and therefore further exploration would be needed if work on this feature progresses.

Examples from user research:

“What sort of information is this? I don’t know? Maybe it’s previous addresses but I don’t know.” from phase 2, round 3

“I don’t understand ‘from the past’. It feels dubious.” from phase 2, round 3

“‘The past’ implies it’s from years ago.” from phase 2, round 3

“Maybe it’s good to review if there was an error in the past.” from phase 2, round 3

The prototype testing flow included a task around ‘connecting’ fictional legacy data and for some, this supported the expectation that data is shared across services via a global push.

Examples from user research:

“DWP would know everything anyway?” from phase 2, round 3

“I should give consent and it automatically updates everything - that’s the whole point.” from phase 2, round 3

We tested a scenario where the task of connecting fictional legacy data presented a useful outcome. User research participants were notified of an ‘address clash’ and were prompted to review their address so that they did not incur a fine from the DVLA. See phase 2, round 3

Users understood this as a possible use of legacy data and identified some benefits.

Examples from user research:

“The implications are that it would flag to DLVA that your address has changed so they might get on your back a bit. I think that has actually happened to me before! It’s helpful because you might not realise that you might get fined if you don’t tell them.” from phase 2, round 5

“This is good - if the info wasn’t linked in the account you could be fined.” from phase 2, round 5"
