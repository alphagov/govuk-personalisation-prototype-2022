Feature prioritisation

As mentioned previously, any features associated with account setup were excluded from the initial service research.

We focused on prioritising features that could be enabled after the account has the capability to save and reuse data.

Top 5 features by value/associated complexity

  1. Trusted data (Medium complexity)
  2. Global push (High complexity)
  3. Notifications (Low complexity)
  4. Reminders (Low complexity)
  5. Use of existing data (Medium complexity)

Although notifications and reminders were regarded as being high value/low complexity, the full value of these will not be realised without the underlying data in place.

Core features vs features that add additional value

  • Without the data led features and the underlying data being in place the full value of other features will be difficult to realise
  • Data led features enable some of the easier features to become a reality

Other significant opportunities

Account flags

Services spoke about there being considerable value in sharing/accessing facts or markers about users.

These account flags fall into two categories:

  • Flags that are visible to the user within the GOV.UK account - Example: “In receipt of PIP” “Deceased”
  • Flags that aren’t visible to the user - Example: “Account has been compromised - fraud risk”

Emerging themes

The operating model

There was concern about the operating model - if the data is wrong somewhere, you have to be able to change it:

  • What is the support mechanism?
  • If the user contacts the contact centre - who's contact centre is this?
  • Will the GOV.UK GDS contact centre need to resolve this or the service the data has come from?
  • How will this be explained to the user?
  • Would the operating model need to be established and functional before some features can be implemented?

User adoption/engagement

Some of the services felt that a number of the proposed features would need to be carefully considered from a user engagement perspective.

Getting users to engage with their GOV.UK account to manage their data and account settings was seen by many as a difficult challenge to overcome in the future - in particular features such as profile completion, notifications and reminders.

User reactions

Exploring how to strike a healthy balance between the government knowing a great deal about the user while also helping them.

Which data is useful data cross government services

  • Some data will only ever be service specific - this raises the question of how useful it would be to save some types of data within the account if it’s unlikely to be reused (it could also become unmanageable if large amounts of data are saved to the account)
  • Some types of data will need to be excluded (for example, special forces personnel)
  • GDPR - Users should only provide information that is necessary (there is a risk of asking for data without an explicit need - such as completing your profile)
  • Awareness of there being a risk around making too many assumptions based on data (derived data)

Data silos and aggregated data stores

Services felt that there could be a risk of GDS creating a data silos

...it really needs to be a personal data store, rather than an aggregated data store yet another GDS silos

Other account set up considerations

Completing ‘hygiene checks’, such as renewing consent permissions periodically.

The methodology we used allowed us to explore features with service teams, learning where they saw value and complexity.

It’s important to remember that the final rankings are opinions and expectations based on a short description of each feature. Much more detailed research is needed to fully understand the true value and complexity of adopting each of the features.

These sessions were conducted with service owners and product managers and the prioritisation exercise was not based on any technical assessments of complexity.
